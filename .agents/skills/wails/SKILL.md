# Wails Desktop Skill (v3 default · v2 legacy)

> **Sources**: Wails v3 docs — <https://v3.wails.io> · Wails v2 docs — <https://wails.io>
> **Status (2026-06)**: **v3 is alpha** (`v3.0.0-alpha.x`, no stable release yet) but
> API-stable and production-used. **v2.x** is the current stable line.
> **This repo's generator (`generador.go`) emits v3 only** — treat v3 as the default;
> reach for v2 syntax only when explicitly working on a v2 codebase.
> For the Go *language* layer (1.26 idioms, slog, context), see the **`go-modern`** skill.

---

## 0 — Detect your version FIRST

Never mix v2 and v3 APIs. Identify the line before writing a single line of Wails code.

| Signal | **v3** | **v2** |
|---|---|---|
| Go import | `github.com/wailsapp/wails/v3/pkg/application` | `github.com/wailsapp/wails/v2` + `.../v2/pkg/runtime` |
| Entry shape | `application.New(...)` → window → `app.Run()` | single `wails.Run(&options.App{...})` |
| Backend exposure | **Services** (`application.NewService`) | **Bind** list of structs |
| Frontend runtime | `@wailsio/runtime` (`Events`, `Window`, `Dialogs`) | `runtime/*` from `wailsjs/runtime` |
| CLI | `wails3` | `wails` |

**Rules**

- When editing files generated by this repo, you are in **v3** — confirm via the import path.
- DO NOT copy v2 snippets (`OnDomReady`, the global `runtime` package) into a v3 project; they
  do not exist there. See §3 for the mapping.

## 1 — Scaffolding & project layout (v3)

v3 **separates** the three phases that v2 fused into one `wails.Run()`:

```go
app := application.New(application.Options{   // 1. create app + register services
    Name:     cfg.DisplayName,
    Services: services,
    Assets:   application.AssetOptions{Handler: application.AssetFileServerFS(assets)},
    ShouldQuit: func() bool { return true },  // intercept quit (unsaved changes, etc.)
    OnShutdown: func() { /* flush, close DB */ },
})
window := app.NewWebviewWindowWithOptions(application.WebviewWindowOptions{ /* ... */ }) // 2.
if err := app.Run(); err != nil { /* ... */ } // 3.
```

**Rules**

- `main.go` is the **only** orchestration point — there is no `app.go` god object (matches
  `tmplMainGo` in `generador.go`). Construct services, then app, then window(s), then `Run()`.
- EMBED the built frontend with `//go:embed all:frontend/dist` and serve via
  `application.AssetFileServerFS`. Keep a placeholder `dist/index.html` so the embed compiles
  before the first `npm run build`.
- USE `application.Options.ShouldQuit`/`OnShutdown` for app-wide lifecycle; use the Service hooks
  (§2) for per-domain lifecycle.

## 2 — Service pattern & lifecycle (v3, default)

A service is a plain struct, registered once and auto-bound to the frontend.

```go
type ExampleService struct{ db *bun.DB }

func NewExampleService(db *bun.DB) *ExampleService { return &ExampleService{db: db} }

func (s *ExampleService) ServiceName() string { return "ExampleService" }                       // optional
func (s *ExampleService) ServiceStartup(ctx context.Context, o application.ServiceOptions) error // optional
func (s *ExampleService) ServiceShutdown() error                                                 // optional

// Registered in main.go:
application.NewService(NewExampleService(bdb))
```

**Rules**

- KEEP services autonomous: inject dependencies through the constructor (`*bun.DB`, a worker, a
  config) — never reach for package globals.
- START long-lived resources (goroutine pools, watchers) in `ServiceStartup` and tear them down in
  `ServiceShutdown`; both run exactly once. Reference impl: `backend/example/service.go`.
- Only **exported** methods are bound. Make the first param `context.Context` for cancellation.

## 3 — Lifecycle mapping v2 → v3 (anti-stale-API guardrail)

The hooks v2 developers reach for **do not exist** in v3. Translate, don't transplant:

| v2 hook (`options.App`) | v3 equivalent |
|---|---|
| `OnStartup(ctx)` | `ServiceStartup(ctx, opts)` (per service) / app construction in `main.go` |
| `OnDomReady(ctx)` | `window.OnWindowEvent(events.Common.WindowRuntimeReady, fn)` |
| `OnBeforeClose(ctx) bool` | `application.Options.ShouldQuit` / `events.Common.WindowClosing` |
| `OnShutdown(ctx)` | `application.Options.OnShutdown` / `ServiceShutdown()` |
| `Bind: []any{...}` | `Services: []application.Service{application.NewService(...)}` |

```go
// ✅ v3 — persist window geometry on close (matches the repo's windowstate handler)
window.OnWindowEvent(events.Common.WindowClosing, func(e *application.WindowEvent) {
    w, h := window.Size(); x, y := window.Position()
    _ = windowstate.Save(context.Background(), bdb, windowstate.State{X: x, Y: y, W: w, H: h})
})
```

**Rule**: in a v3 project, NEVER write `OnDomReady`/`OnBeforeClose` — use window events instead.

## 4 — Bindings & type-safety

`wails3 generate bindings` reads exported service methods/structs via AST and writes JS/TS wrappers
to `frontend/bindings/<go-import-path>/` (auto-generated, gitignored).

```bash
wails3 generate bindings        # JS bindings
wails3 generate bindings -ts     # + TypeScript type definitions
```

**Rules**

- RE-GENERATE after changing any exported signature or bound struct. `wails3 dev` watches Go files
  and regenerates automatically — restart it if a new service/method isn't picked up.
- AVOID generics and exotic types on bound signatures. Return **concrete Result structs** for
  business outcomes (matches `tmplExampleResult`): `{ Data *T; Error *shared.AppError }`.
- NEVER import from `frontend/bindings/` directly across the app — wrap each service in **one**
  module and import that everywhere (keeps the gitignored, regenerated path swappable).
- TYPES that don't marshal cleanly to JS (e.g. monetary `Decimal`) must serialize as **strings** —
  keep them strings on the frontend and format with `Intl.NumberFormat`, never `parseFloat` for math.

## 5 — Native runtime & no-UI-freeze guardrails

| Capability | **v3** | **v2** |
|---|---|---|
| Emit event (Go) | `window.EmitEvent("name", data)` | `runtime.EventsEmit(ctx, "name", data)` |
| Window/dialog/clipboard (JS) | `@wailsio/runtime`: `Window`, `Dialogs`, `Clipboard`, `Screens` | `wailsjs/runtime` |
| Set title (Go) | window method on `*application.WebviewWindow` | `runtime.WindowSetTitle(ctx, ...)` |

**Rules**

- A bound method runs on a call handler — **NEVER block it on long work.** Propagate the incoming
  `context.Context`, hand heavy work to a goroutine (or the repo's `background.Worker`), and stream
  results back via emitted events. Honor `ctx.Done()` to support cancellation/shutdown.
- DO NOT spawn naked goroutines for app-lifetime work; own them in a service so `ServiceShutdown`
  can drain them (prevents goroutine leaks — cross-check the `go-modern` goroutine-leak profile).
- Treat the frontend as untrusted input: validate arguments in Go before touching the DB/filesystem.

---

## 6 — Error handling across the Go↔JS bridge

- **System errors** (DB/file I/O, programmer faults): return a native Go `error` → Wails **rejects**
  the TypeScript promise. Let it propagate.
- **Business errors** (validation, not-found, conflict): return a concrete Result with
  `*shared.AppError` set → the promise **resolves**; the frontend inspects `result.error`. Codes
  live in `backend/shared/errors.go`.

Use the `go-modern` skill for the Go-side error idioms (`errors.AsType`, wrapping).

## 7 — Command flows

```bash
# Dev (hot reload + binding regen)
wails3 dev                      # v3   │   wails dev    # v2

# Data-race debugging (Go backend)
go test -race ./...
go build -race -o bin/app-dev . # race-instrumented dev binary

# Production build
wails3 build                    # v3   │   wails build  # v2
# Frontend is minified by its own build step: tsc --noEmit && vite build
```

**Cross-compilation (v3).** Build on native runners per OS (Taskfile targets `darwin:build` /
`windows:build` / `linux:build`; CI uses a `GOOS` matrix over `ubuntu/macos/windows-latest`). macOS
needs `MACOSX_DEPLOYMENT_TARGET` + matching `CGO_CFLAGS/CGO_LDFLAGS`. Full matrix, signing, and a
GitHub Actions workflow are in [REFERENCE.md](./REFERENCE.md).

**Rules**

- PIN the Wails version (v3 is alpha; nightly releases can break) — edit `go.mod` and run
  `go mod tidy`, then `wails3 doctor` to verify the toolchain.
- RUN `wails3 doctor` after any toolchain/dependency change before reporting a build as working.
